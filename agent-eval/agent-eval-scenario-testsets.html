线上 Agent 业务，如何按场景设计评测类型和评测集

设计线上 AI Agent 评测体系时，不要先问“做人评还是机评”，而要先问：这次评测是被什么业务场景触发的。

很多团队做 Agent 评测时，第一反应是讨论评测手段：要不要端到端，要不要测工具，要不要上自动 judge，要不要做人评抽检。 这些问题都重要，但它们不应该是最外层分类。更靠前的问题是：这次评测为什么发生。

一个真实线上 Agent 业务，通常至少会遇到四类场景：新能力上线、线上质量巡检、模型或工具链路变更、用户负反馈治理。 这四类场景的目标不同，样本来源不同，输出结果也不同。如果混在一起，评测集很容易变成一个“大杂烩”。

先按业务场景确定评测类型，再决定评测集、评测方法和输出结论。

先按场景分，而不是先按方法分

线上 Agent 评测最容易混淆的是“评测类型”和“评测方法”。人评、机评、端到端评测、工具评测、GUI 评测，都是执行方法； 上线准入、周期巡检、专项变更、负反馈治理，才更像业务视角下的评测类型。

如果不先分清触发场景，常见问题会很快出现：

  • 用上线准入的理想样本去判断线上质量，结果看不到真实用户问题。
  • 用线上 bad case 去判断新能力能不能上线，样本又太偏，不能代表需求覆盖。
  • 只看最终回答，不看 Trace、工具调用和端侧展示，问题很难归因。
  • 只形成评测报告，不进入修复和回归，评测对迭代没有约束力。

四类常见场景

我更倾向于把线上 Agent 评测先分成下面四类。它们不是互斥的，但每一类的首要问题不一样。

业务场景 评测类型 核心问题 适合的评测集 常用方法 输出结果
新需求、新能力上线前 需求评测 / 准入评测 核心链路是否达标，能不能上线 需求评测集、理想集、必过集、边界集 端到端评测、人评抽检、机评批跑、工具链路检查 上线准入结论、阻塞问题、风险项
业务已经在线上运行 周期评测 / 质量巡检 线上质量是否稳定，有没有退化 线上真实 Query、黄金集、趋势样本、竞品样本 定期抽样、人机结合、质量趋势分析、bad case 标注 质量趋势、问题分布、版本对比、巡检报告
模型、工具、GUI、策略或工程链路变更 专项评测 / 变更评测 变更有没有引入新风险 专项集、影响面样本、黄金回归集、历史 bad case 集 变更前后对比、Trace diff、工具原子评测、端侧回归 风险结论、回滚建议、灰度建议、回归结果
用户负反馈、投诉、低满意度、异常行为 线上负反馈治理 问题为什么发生，谁来修，怎么防复发 Bad case 集、负反馈样本、问题簇样本、复现样本 机评初筛、人评复核、证据核验、聚类归因、修复后重试 责任域、问题簇、修复任务、回归结论

评测集要按用途分层

评测集最怕所有样本混在一起。上线前、上线后、专项变更、负反馈治理，对样本的要求完全不同。 一个更稳的做法，是按用途维护多层评测集。

理想集

理想集来自 PRD、产品预期、专家经验或人工构造。它描述的是“这个能力理想状态应该怎么表现”。 它适合需求冷启、拆 rubric 和统一验收口径，但不能单独代表真实线上分布。

必过集

必过集覆盖最核心、最不能失败的场景，主要用来守住上线底线。它适合做上线准入门槛，也适合放进自动回归。 这类样本不需要很多，但每一条都应该有明确的阻断意义。

黄金集

黄金集是一批长期稳定维护的高价值样本，用来做版本间可比的质量观察。 它适合周期评测和版本回归，但要控制更新频率，否则历史分数会失去可比性。

Bad case 集

Bad case 集来自线上负反馈、人评发现、历史事故和专项测试。它的价值不是代表整体质量，而是防止问题复发。 这类样本要尽量保留现场证据，不能只存 Query 和 Response。

挑战集

挑战集覆盖多约束、多轮、长上下文、组合意图和边界条件，主要用来看能力上限。 它适合模型升级前后的能力压测，但不适合代表线上整体质量。

专项集

专项集围绕某次模型、工具、GUI 或策略变更构建，样本要贴近变更影响面。 它可以复用黄金集和历史 bad case，但关键不是样本数量，而是能不能命中风险点。

评测集和场景怎么匹配

不同评测集可以复用，但不要混用。下面这张表可以作为设计评测任务时的检查清单。

评测集类型 最适合的场景 构建方式 维护重点
需求评测集 上线前准入 从 PRD、理想样例、核心流程、边界条件中构建 覆盖需求点，区分必须满足和可优化项
必过集 上线前阻断、核心链路回归 从核心意图、高频入口、历史高优问题中挑选 宁少勿滥，保证每条都有阻断意义
黄金集 周期评测、版本对比 从真实线上 Query、核心业务场景、稳定样本中沉淀 保持稳定，控制更新节奏，保证长期可比
挑战集 复杂能力评估、模型能力压测 多约束、多轮、长上下文、边界条件、组合意图 更适合看能力上限，不适合代表线上整体质量
专项集 模型、工具、GUI、策略变更 围绕变更影响面构造，复用黄金集和 bad case 集 样本要贴着变更点，避免大水漫灌
Bad case 集 线上负反馈治理、修复回归 从负反馈、投诉、低满意度、人工巡检中落库 保留 Trace、版本、时间戳、工具结果和责任域

小结

线上 Agent 评测更像一套按业务场景路由的质量系统：上线前看准入，上线后看稳定，技术变更看风险，线上负反馈看治理。 每类评测都要配自己的评测集、评测方法和后续动作。

先把场景分清楚，后面的人评、机评、端到端、工具评测、GUI 评测才有位置。否则评测集会越攒越多，但很难回答最关键的问题： 这次评测到底要支持哪个业务决策。
